# 01-1. CRD 파일 한 줄씩 이해하기

## 먼저 알아야 할 것: Native Resources vs Custom Resources

Kubernetes에는 두 종류의 리소스가 있다.

|               | Native Resources (기본 리소스)  | Custom Resources (커스텀 리소스)   |
| ------------- | ------------------------------- | ---------------------------------- |
| 누가 만들었나 | Kubernetes에 원래 내장되어 있음 | 사용자가 직접 정의해서 추가        |
| 예시          | Pod, Service, Deployment        | 우리가 만든 Snack, SimpleApp       |
| 구조          | 미리 정해져 있음                | 사용자가 schema로 직접 정의        |
| 용도          | Kubernetes 핵심 기능            | 도메인별 맞춤 기능 (DB, 인증서 등) |

비유:

- Native Resources = 학교에 원래 있는 시설 (교실, 급식실, 도서관)
- Custom Resources = 우리가 새로 만든 시설 (간식 신청 창구)

**CRD는 Custom Resource를 등록하기 위한 "신청 양식"이다.**
CRD를 등록하면, 그때부터 그 종류의 Custom Resource를 만들 수 있다.

---

`examples/crd.yaml` 파일을 한 줄씩 뜯어본다.

## 전체 구조

CRD 파일은 크게 3 부분이다:

```text
1. 이 파일 자체가 무엇인가 (apiVersion, kind, metadata)
2. 새 리소스의 이름과 범위 (group, names, scope)
3. 새 리소스의 모양 (schema)
```

## 1부: 이 파일 자체가 무엇인가

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: snacks.school.example.com
```

### apiVersion: apiextensions.k8s.io/v1

"이 YAML은 Kubernetes의 CRD 기능을 사용한다"는 뜻이다.
Kubernetes에 원래 내장된 API이다. 우리가 만드는 것이 아니다.
현재 무조건 `apiextensions.k8s.io/v1`이다. 다른 선택지 없음 (고정값).

모든 Kubernetes YAML에는 apiVersion이 있다:

- Pod: `v1`
- Deployment: `apps/v1`
- CRD: `apiextensions.k8s.io/v1`

### kind: CustomResourceDefinition

"이 YAML은 CRD를 등록하는 것이다"라는 뜻이다.
Deployment를 만들 때 `kind: Deployment`라고 쓰는 것과 같다.

### metadata.name: snacks.school.example.com

CRD 자체의 이름이다. Kubernetes 안에서 이 CRD를 식별하는 고유 이름.
**반드시 `{plural}.{group}` 형식**이어야 한다.

```text
snacks.school.example.com
^^^^^^ ^^^^^^^^^^^^^^^^^^
plural   group
```

- `snacks` = 아래 `spec.names.plural`에서 정의한 값
- `school.example.com` = 아래 `spec.group`에서 정의한 값

마음대로 이름을 지으면 (예: `my-snack-crd`) Kubernetes가 CRD 등록을 거부한다.

## 2부: 새 리소스의 이름과 범위

```yaml
spec:
  group: school.example.com
  names:
    kind: Snack
    plural: snacks
    singular: snack
  scope: Namespaced
```

### group: school.example.com

API 그룹 이름이다. CR을 만들 때 `apiVersion`에 들어간다:

```yaml
apiVersion: school.example.com/v1
  ^^^^^^^^^^^^^^^^^^
  여기가 group
```

실제 도메인을 소유할 필요는 없다. 이름 충돌을 방지하기 위한 네임스페이스 역할이다.

### names

| 필드     | 값     | 어디서 사용되나                                |
| -------- | ------ | ---------------------------------------------- |
| kind     | Snack  | CR의 YAML에서: `kind: Snack`                   |
| plural   | snacks | kubectl에서: `kubectl get snacks`              |
| singular | snack  | kubectl에서: `kubectl get snack` (단수도 가능) |

kind는 PascalCase(대문자 시작), plural/singular는 소문자.

### scope: Namespaced

이 리소스가 namespace에 속하는지 여부.

- `Namespaced`: `default`, `kube-system` 등 특정 namespace에 속한다 (대부분의 리소스)
- `Cluster`: 클러스터 전체에 하나. namespace 없음 (Node, PersistentVolume처럼)

## 3부: 새 리소스의 모양 (스키마)

```yaml
versions:
  - name: v1
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            properties:
              menu:
                type: string
              quantity:
                type: integer
```

### versions

**주의: CRD 파일에는 "version"이 2개 있다. 헷갈리지 말 것.**

| 어떤 version              | 위치             | 누가 정하나       | 값                                  |
| ------------------------- | ---------------- | ----------------- | ----------------------------------- |
| `apiextensions.k8s.io/v1` | 맨 위 apiVersion | Kubernetes가 정함 | 무조건 `v1` (고정)                  |
| `spec.versions[].name`    | 여기             | 우리가 정함       | `v1alpha1`, `v1beta1`, `v1` 등 자유 |

맨 위 apiVersion은 "CRD 기능 자체의 버전"이고, 여기 versions는 "우리가 만드는 리소스의 버전"이다.

하나의 CRD가 여러 버전을 가질 수 있다 (v1alpha1, v1beta1, v1 등).
지금은 v1 하나만 있다.

- `served: true` -- 이 버전을 API에서 사용할 수 있다
- `storage: true` -- 이 버전으로 etcd에 저장한다 (여러 버전 중 하나만 true 가능)

### schema (openAPIV3Schema)

**"이 리소스의 spec에 어떤 필드가 들어갈 수 있는가"**를 정의한다.

```yaml
spec: # CR의 spec 부분
  properties:
    menu: # 필드 이름: menu
      type: string # 타입: 문자열
    quantity: # 필드 이름: quantity
      type: integer # 타입: 정수
```

이것 덕분에 CR을 만들 때:

```yaml
spec:
  menu: chocopie # string이니까 OK
  quantity: 30 # integer이니까 OK
  color: red # schema에 없는 필드 -- 무시됨
```

잘못된 타입을 넣으면 거부된다:

```yaml
spec:
  quantity: "서른" # integer인데 string을 넣음 -- 에러!
```

## CRD와 CR의 관계 한눈에 보기

```text
CRD (crd.yaml)                    CR (cr.yaml)
--------------                    ------------
group: school.example.com    -->  apiVersion: school.example.com/v1
kind: Snack                  -->  kind: Snack
schema:
  menu: string               -->  spec:
  quantity: integer           -->    menu: chocopie
                                     quantity: 30
```

CRD가 "틀"이고, CR은 그 "틀에 맞춰 작성한 데이터"이다.
