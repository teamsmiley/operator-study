# 04. 빌드와 배포

## make 명령어 흐름

```text
types.go 수정
    |
    v
make generate    --> DeepCopy Go 코드 자동 생성
    |
    v
make manifests   --> CRD YAML + RBAC YAML 자동 생성
    |
    v
make install     --> CRD를 클러스터에 등록
    |
    v
make run         --> Controller를 로컬에서 실행
```

## 각 명령어 상세 설명

### make generate -- DeepCopy 코드 자동 생성

```text
입력: api/v1/simpleapp_types.go (우리가 작성한 struct)
출력: api/v1/zz_generated.deepcopy.go (자동 생성된 코드)
```

Kubernetes 내부에서 오브젝트를 복사할 때 필요한 DeepCopy 함수를 자동 생성한다.
모든 Operator에 필요하지만, 매번 직접 쓰면 반복 작업이라서 자동화한 것이다.
`zz_` 접두사 = "자동 생성 파일, 직접 수정하지 마라"는 관례.

### make manifests -- CRD YAML + RBAC YAML 생성

```text
입력 1: api/v1/simpleapp_types.go (struct + kubebuilder marker)
출력 1: config/crd/bases/apps.example.com_simpleapps.yaml (CRD YAML)
        --> 01장에서 직접 작성한 crd.yaml과 같은 것을 Go 코드에서 자동 생성

입력 2: controller.go의 RBAC marker 주석
출력 2: config/rbac/role.yaml (권한 설정)
```

01장에서 Snack crd.yaml을 직접 YAML로 작성했는데,
kubebuilder는 Go 코드에서 YAML을 자동 생성해준다.

### make install -- CRD를 클러스터에 등록

```text
내부적으로 하는 일:
  make manifests (자동 실행) --> kubectl apply -f config/crd/bases/
```

01장에서 `kubectl apply -f crd.yaml` 했던 것과 같다.
이것을 하지 않으면 CR 생성 시 "SimpleApp이 뭔지 모름" 에러가 난다.

### make run -- Controller를 로컬에서 실행 (개발용)

```text
내부적으로 하는 일:
  make manifests + make generate (자동 실행) --> go run ./cmd/main.go
```

02장에서 배운 Watch + Reconcile이 시작된다.
터미널을 점유하며, CR 변경을 감시하다가 Reconcile을 실행한다.

**참고: make run과 make install은 내부적으로 generate/manifests를 자동 실행한다.
따라서 types.go를 수정한 후 make run만 해도 전부 자동으로 처리된다.**

### 자동 실행 의존성 정리

| 명령 | 자동으로 포함하는 것 |
|------|---------------------|
| make install | manifests |
| make run | manifests + generate + fmt + vet |
| make build | manifests + generate + fmt + vet |

## make run의 동작 원리

`make run`은 **클러스터 안에 아무것도 설치하지 않는다.**
내 PC에서 프로세스로 돌면서, kubeconfig을 통해 API Server와 통신한다.

```text
내 PC (make run)                          k3d 클러스터
----------------                          ------------
Controller 프로세스                        API Server
  |                                           |
  |-- Watch 요청 (HTTP long-poll) ----------->|  "SimpleApp 변경 알려줘"
  |                                           |
  |<-- 이벤트 수신 ----------------------------|  "my-app 생성됨!"
  |                                           |
  |  Reconcile 실행                           |
  |  (Deployment 오브젝트 조립)               |
  |                                           |
  |-- Create 요청 (HTTP POST) --------------->|  "이 Deployment 만들어줘"
  |                                           |
  |                                        API Server가 Deployment 저장
  |                                        kubelet이 Pod 생성
```

`kubectl get pods`로 확인해도 Controller Pod는 보이지 않는다.
내 PC에서 원격으로 일하고 있을 뿐이다.

### :8081 포트는 뭔가?

`make run`을 두 번 하면 `:8081 포트 충돌` 에러가 난다.
이 포트는 k8s API 통신용이 아니라, **health probe (살아있나 확인)** 용이다.

```text
:8081 포트        = "나 살아있어?" 확인용 (health check)
kubeconfig 주소   = k8s API Server와 실제 통신 (Watch, Create, Update)
```

## 운영 배포 (make deploy)

`make run`은 개발용이다. **실제 운영에서는 Controller를 Docker 이미지로 만들어서 클러스터 안에 Pod로 배포**해야 한다.

### 배포 순서

```bash
# 1. Docker 이미지 빌드
make docker-build IMG=myoperator:v0.1.0

# 2. 이미지를 레지스트리에 push (또는 k3d에서 로컬 import)
make docker-push IMG=myoperator:v0.1.0
# k3d 로컬: k3d image import myoperator:v0.1.0 -c operator-lab

# 3. 클러스터에 배포
make deploy IMG=myoperator:v0.1.0
```

### make deploy가 하는 일 (전부 한 번에)

```text
make deploy IMG=myoperator:v0.1.0
    |
    v
1. CRD 설치 (make install이 하던 것)
2. Controller Pod 배포 (make run이 하던 것을 Pod로)
3. RBAC 설정 (ServiceAccount, ClusterRole 등)
4. 전부 myoperator-system namespace에 생성
```

개발 모드에서는 2단계였던 것이 운영 모드에서는 1단계로 줄어든다:

```text
개발:  make install (CRD) + make run (Controller 로컬 실행)   --> 2단계
운영:  make deploy (CRD + Controller Pod + RBAC 한 번에)      --> 1단계
```

### make run vs make deploy 비교

|           | make run (개발)               | make deploy (운영)            |
| --------- | ----------------------------- | ----------------------------- |
| 실행 위치 | 내 PC (원격 근무)             | 클러스터 안 Pod (출근)        |
| 터미널    | 점유함                        | 점유 안 함                    |
| 재시작    | 수동 (Ctrl+C, 다시 실행)      | 자동 (Pod crash -> 자동 복구) |
| Pod 보임? | kubectl get pods에 안 보임    | kubectl get pods에 보임       |
| 비유      | 집에서 원격 근무하는 선생님   | 학교에 출근한 선생님          |

## manifest 파일 위치

`make deploy`가 적용하는 manifest들은 `config/` 폴더에 있다:

```text
config/
  crd/bases/
    apps.example.com_simpleapps.yaml    # CRD (리소스 종류 등록)
  rbac/
    role.yaml                           # RBAC (Controller 권한)
    role_binding.yaml
    service_account.yaml
  manager/
    manager.yaml                        # Controller Deployment (Pod 배포)
```

`make manifests`가 Go 코드에서 이 YAML들을 자동 생성하고,
`make deploy`가 kustomize로 합쳐서 한 번에 `kubectl apply` 한다.

## 접근 불가능한 클러스터에 배포하려면

`make deploy`는 내 PC에서 kubeconfig으로 접근 가능한 클러스터에만 쓸 수 있다.
접근 불가능한 클러스터에는 manifest 파일을 전달해야 한다:

```bash
# 배포용 YAML 하나로 합치기 (CRD + Controller + RBAC 전부 포함)
make build-installer IMG=ghcr.io/teamsmiley/myoperator:v0.1.0

# 결과: dist/install.yaml 생성
```

이 파일을 대상 클러스터에 접근 가능한 사람에게 전달하면:

```bash
kubectl apply -f install.yaml   # 이 한 줄로 Operator 전체 설치
```

결국 배포에 필요한 것은 **이미지(레지스트리) + manifest(YAML)** 두 가지뿐이다.

## make deploy 후 확인하는 법

```bash
# Controller Pod 확인
kubectl get pods -n myoperator-system

# Controller 로그 보기 (make run 터미널 대신)
kubectl logs -n myoperator-system -l control-plane=controller-manager -f

# 정리할 때
make undeploy
```
