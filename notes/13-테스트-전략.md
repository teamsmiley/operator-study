# 테스트 전략

## 한 줄 요약

Operator 테스트는 세 단계로 나뉜다: Unit(단위), Integration(통합), E2E(종단간).
kubebuilder가 세 종류의 테스트 파일을 모두 scaffolding해 준다.

---

## 비유로 이해하기

자동차를 만들었다고 하자.

| 테스트 종류 | 자동차 비유                     | Operator에서는                            |
| ----------- | ------------------------------- | ----------------------------------------- |
| Unit        | 엔진만 따로 돌려본다            | 함수 하나를 격리해서 테스트               |
| Integration | 엔진 + 변속기를 연결해서 테스트 | 가짜 K8s API 서버와 함께 Reconcile 테스트 |
| E2E         | 실제 도로에서 운전해 본다       | 진짜 클러스터에 배포하고 전체 흐름 테스트 |

---

## 테스트 프레임워크

kubebuilder는 **Ginkgo + Gomega** 조합을 사용한다.

| 프레임워크 | 역할                     | Go 표준과 비교   |
| ---------- | ------------------------ | ---------------- |
| Ginkgo     | BDD 스타일 테스트 실행기 | `testing.T` 대체 |
| Gomega     | 가독성 좋은 Assertion    | `assert` 대체    |

```go
// Ginkgo + Gomega 스타일
var _ = Describe("SimpleApp Controller", func() {
    It("should successfully reconcile", func() {
        Expect(err).NotTo(HaveOccurred())
    })
})

// Go 표준 스타일 (비교용)
func TestReconcile(t *testing.T) {
    if err != nil {
        t.Fatalf("unexpected error: %v", err)
    }
}
```

## 1. Integration 테스트 (envtest)

### 우리 프로젝트의 파일

```
internal/controller/
    suite_test.go                   -- 테스트 환경 설정 (envtest 시작/종료)
    simpleapp_controller_test.go    -- 실제 테스트 케이스
```

### envtest란?

**가짜 쿠버네티스 API 서버**를 로컬에서 띄워서 테스트하는 도구다.
진짜 클러스터 없이도 CRD 등록, CR 생성, Reconcile 호출이 가능하다.

```
envtest 환경:
    etcd (내장)  +  kube-apiserver (내장)
        ↑
    테스트 코드에서 k8sClient로 접근
```

실제 클러스터와의 차이:

| 있는 것   | 없는 것        |
| --------- | -------------- |
| API 서버  | kubelet        |
| etcd      | 실제 Pod 실행  |
| CRD 등록  | 네트워크, 볼륨 |
| CRUD 동작 | 스케줄링       |

### suite_test.go -- 환경 설정

```go
// 테스트 시작 전: envtest 환경을 띄운다
var _ = BeforeSuite(func() {
    // 1. 로거 설정
    logf.SetLogger(zap.New(zap.WriteTo(GinkgoWriter), zap.UseDevMode(true)))

    // 2. envtest 환경 생성 (가짜 API 서버 + CRD 로드)
    testEnv = &envtest.Environment{
        CRDDirectoryPaths:     []string{filepath.Join("..", "..", "config", "crd", "bases")},
        ErrorIfCRDPathMissing: true,
    }

    // 3. 환경 시작 → API 서버 주소를 받아온다
    cfg, err = testEnv.Start()

    // 4. 클라이언트 생성
    k8sClient, err = client.New(cfg, client.Options{Scheme: scheme.Scheme})
})

// 테스트 끝: 환경 정리
var _ = AfterSuite(func() {
    cancel()
    testEnv.Stop()
})
```

### simpleapp_controller_test.go -- 테스트 케이스

```go
var _ = Describe("SimpleApp Controller", func() {
    Context("When reconciling a resource", func() {

        BeforeEach(func() {
            // CR 생성
            resource := &appsv1.SimpleApp{
                ObjectMeta: metav1.ObjectMeta{
                    Name:      "test-resource",
                    Namespace: "default",
                },
            }
            Expect(k8sClient.Create(ctx, resource)).To(Succeed())
        })

        AfterEach(func() {
            // CR 정리
            Expect(k8sClient.Delete(ctx, resource)).To(Succeed())
        })

        It("should successfully reconcile the resource", func() {
            // Reconciler를 직접 호출한다
            controllerReconciler := &SimpleAppReconciler{
                Client: k8sClient,
                Scheme: k8sClient.Scheme(),
            }

            _, err := controllerReconciler.Reconcile(ctx, reconcile.Request{
                NamespacedName: typeNamespacedName,
            })
            Expect(err).NotTo(HaveOccurred())
        })
    })
})
```

핵심: **Reconcile 함수를 직접 호출**해서 에러가 없는지 확인한다.
실제 Manager를 띄우는 게 아니라, 함수 하나만 호출하는 것이다.

### 실행 방법

```bash
# envtest 바이너리 설치 (최초 1회)
make setup-envtest

# Integration 테스트 실행
make test
```

## 2. Webhook 테스트

### 우리 프로젝트의 파일

```
internal/webhook/v1/
    webhook_suite_test.go     -- Webhook 테스트 환경 설정
    simpleapp_webhook_test.go -- Webhook 테스트 케이스
```

### 현재 상태

kubebuilder가 scaffolding한 상태 그대로다.
TODO 주석만 있고, 실제 테스트 로직은 아직 작성되지 않았다.

```go
Context("When creating SimpleApp under Defaulting Webhook", func() {
    // TODO (user): Add logic for defaulting webhooks
})

Context("When creating or updating SimpleApp under Validating Webhook", func() {
    // TODO (user): Add logic for validating webhooks
})
```

### 작성한다면 이런 식

```go
It("Should apply default replicas when not specified", func() {
    obj.Spec.Replicas = nil
    defaulter.Default(ctx, obj)
    Expect(*obj.Spec.Replicas).To(Equal(int32(1)))
})

It("Should deny creation if image is empty", func() {
    obj.Spec.Image = ""
    _, err := validator.ValidateCreate(ctx, obj)
    Expect(err).To(HaveOccurred())
})

It("Should deny replicas over 10", func() {
    replicas := int32(11)
    obj.Spec.Replicas = &replicas
    obj.Spec.Image = "nginx:latest"
    _, err := validator.ValidateCreate(ctx, obj)
    Expect(err).To(HaveOccurred())
})
```

## 3. E2E 테스트

### 우리 프로젝트의 파일

```
test/e2e/
    e2e_suite_test.go  -- E2E 테스트 스위트 설정
    e2e_test.go        -- 실제 E2E 테스트
```

### 특징

- `//go:build e2e` 태그가 있어서 일반 `go test`로는 실행되지 않는다
- **진짜 클러스터**에 CRD 설치, Operator 배포, 전체 동작을 검증한다
- `kubectl` 명령어를 직접 실행한다

### 테스트 흐름

```
BeforeAll:
    1. kubectl create ns myoperator-system
    2. make install         (CRD 설치)
    3. make deploy          (Operator 배포)

테스트:
    4. Operator Pod가 Running인지 확인
    5. 메트릭 엔드포인트가 동작하는지 확인
    6. cert-manager가 인증서를 생성했는지 확인
    7. Webhook CA가 주입되었는지 확인

AfterAll:
    8. make undeploy        (Operator 제거)
    9. make uninstall       (CRD 제거)
   10. kubectl delete ns    (Namespace 제거)
```

### 실행 방법

```bash
# E2E 테스트 실행 (진짜 클러스터 필요)
make test-e2e
```

## 세 가지 테스트 비교

|                | Integration (envtest)  | Webhook        | E2E                |
| -------------- | ---------------------- | -------------- | ------------------ |
| 클러스터 필요? | 아니오 (가짜 API 서버) | 아니오         | 예 (진짜 클러스터) |
| 속도           | 빠름 (초 단위)         | 빠름 (초 단위) | 느림 (분 단위)     |
| 테스트 범위    | Reconcile 로직         | Webhook 로직   | 전체 배포 + 동작   |
| CI에서         | 항상 실행              | 항상 실행      | 선택적 실행        |
| make 명령      | `make test`            | `make test`    | `make test-e2e`    |

---

## 테스트 피라미드

```
        /\
       /  \         E2E (적게, 느리지만 확실)
      / E2E\
     /------\
    /        \      Integration (중간)
   /Integration\
  /------------\
 /              \   Unit (많이, 빠르게)
/   Unit Tests   \
------------------
```

- Unit: 가장 많이, 가장 빠르게 (순수 함수 테스트)
- Integration: 핵심 흐름 검증 (envtest로 Reconcile 테스트)
- E2E: 배포 전 최종 확인 (진짜 클러스터에서 전체 동작)

## 실행 명령어 정리

```bash
# Integration + Webhook 테스트 (envtest 사용, 클러스터 불필요)
make test

# E2E 테스트 (진짜 클러스터 필요)
make test-e2e

# 특정 테스트만 실행
go test ./internal/controller/... -v

# Ginkgo로 직접 실행 (더 상세한 출력)
ginkgo -v ./internal/controller/...
```

## 정리

```
make test (로컬에서 빠르게)
├── Integration 테스트 (envtest)
│   └── Reconcile 함수를 직접 호출해서 검증
└── Webhook 테스트
    └── Defaulter/Validator 함수를 직접 호출해서 검증

make test-e2e (클러스터에서 확실하게)
└── E2E 테스트
    └── 진짜 배포 후 kubectl로 전체 동작 검증
```

핵심: kubebuilder가 세 종류 테스트 파일을 전부 scaffolding해 준다.
개발자는 TODO 주석 부분에 자기 로직에 맞는 테스트를 채워넣으면 된다.
