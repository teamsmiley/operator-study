# 테스트 전략

## 한 줄 요약

Operator 테스트는 세 단계로 나뉜다: Unit(단위), Integration(통합), E2E(종단간).
kubebuilder가 Integration과 E2E 테스트 파일을 scaffolding해 준다.

---

## 비유로 이해하기

자동차를 만들었다고 하자.

| 테스트 종류 | 자동차 비유                     | Operator에서는                            |
| ----------- | ------------------------------- | ----------------------------------------- |
| Unit        | 엔진만 따로 돌려본다            | 함수 하나를 격리해서 테스트               |
| Integration | 엔진 + 변속기를 연결해서 테스트 | 가짜 K8s API 서버와 함께 Reconcile 테스트 |
| E2E         | 실제 도로에서 운전해 본다       | 진짜 클러스터에 배포하고 전체 흐름 테스트 |

---

## 테스트 프레임워크

kubebuilder는 **Ginkgo + Gomega** 조합을 사용한다.

| 프레임워크 | 역할                     | Go 표준과 비교   |
| ---------- | ------------------------ | ---------------- |
| Ginkgo     | BDD 스타일 테스트 실행기 | `testing.T` 대체 |
| Gomega     | 가독성 좋은 Assertion    | `assert` 대체    |

```go
// Ginkgo + Gomega 스타일
var _ = Describe("SimpleApp Controller", func() {
    It("should successfully reconcile", func() {
        Expect(err).NotTo(HaveOccurred())
    })
})

// Go 표준 스타일 (비교용)
func TestReconcile(t *testing.T) {
    if err != nil {
        t.Fatalf("unexpected error: %v", err)
    }
}
```

## 1. Unit 테스트

### kubebuilder는 Unit 테스트를 scaffolding하지 않는다

scaffolding된 테스트 파일을 보면 Unit 테스트가 없다.
이유는 Operator 코드의 특성 때문이다.

Reconcile 함수의 대부분은 K8s API 호출이다:

```go
r.Get(ctx, ...)      // API 서버에서 읽기
r.Create(ctx, ...)   // API 서버에 생성
r.Update(ctx, ...)   // API 서버에 업데이트
```

순수 로직이 적고 API 의존이 많기 때문에,
**envtest Integration 테스트가 Operator의 사실상 Unit 테스트 역할**을 한다.

### 헬퍼 함수 테스트

`buildDeployment()` 같은 헬퍼 함수는 Unit 테스트 대상이 된다.
API 호출 없이 입력 -> 출력만 검증하면 되기 때문이다.

```go
func TestBuildDeployment(t *testing.T) {
    replicas := int32(3)
    app := &appsv1.SimpleApp{
        ObjectMeta: metav1.ObjectMeta{
            Name:      "myapp",
            Namespace: "default",
        },
        Spec: appsv1.SimpleAppSpec{
            Image:    "nginx:1.25",
            Replicas: &replicas,
        },
    }

    r := &SimpleAppReconciler{}
    deploy := r.buildDeployment(app)

    // 이름이 맞는지
    if deploy.Name != "myapp" {
        t.Errorf("expected name myapp, got %s", deploy.Name)
    }
    // 이미지가 맞는지
    if deploy.Spec.Template.Spec.Containers[0].Image != "nginx:1.25" {
        t.Errorf("expected image nginx:1.25")
    }
    // replicas가 맞는지
    if *deploy.Spec.Replicas != 3 {
        t.Errorf("expected 3 replicas, got %d", *deploy.Spec.Replicas)
    }
}
```

### Webhook 함수 테스트

Webhook의 Defaulter/Validator도 함수를 직접 호출하므로 Unit 테스트 성격이다.
API 서버 없이, envtest 없이 실행 가능하다.

관련 커밋: `b4fc5cf` (Webhook 테스트 6개 추가, 커버리지 0% -> 86.7%)

테스트 내용:

```go
It("Should apply default replicas when not specified", func() {
    obj.Spec.Replicas = nil
    defaulter.Default(ctx, obj)
    Expect(*obj.Spec.Replicas).To(Equal(int32(1)))
})

It("Should deny creation if image is empty", func() {
    obj.Spec.Image = ""
    _, err := validator.ValidateCreate(ctx, obj)
    Expect(err).To(HaveOccurred())
})

It("Should deny replicas over 10", func() {
    replicas := int32(11)
    obj.Spec.Replicas = &replicas
    obj.Spec.Image = "nginx:latest"
    _, err := validator.ValidateCreate(ctx, obj)
    Expect(err).To(HaveOccurred())
})
```

### Unit 테스트는 Go 표준 `testing` 패키지만으로 충분

```bash
go test ./internal/controller/... -v
go test ./internal/webhook/... -v
```

## 2. Integration 테스트 (envtest)

### 우리 프로젝트의 파일

```
internal/controller/
    suite_test.go                   -- 테스트 환경 설정 (envtest 시작/종료)
    simpleapp_controller_test.go    -- 실제 테스트 케이스
```

### envtest란?

**가짜 쿠버네티스 API 서버**를 로컬에서 띄워서 테스트하는 도구다.
진짜 클러스터 없이도 CRD 등록, CR 생성, Reconcile 호출이 가능하다.

```text
envtest 환경:
    etcd (내장)  +  kube-apiserver (내장)
        ↑
    테스트 코드에서 k8sClient로 접근
```

실제 클러스터와의 차이:

| 있는 것   | 없는 것        |
| --------- | -------------- |
| API 서버  | kubelet        |
| etcd      | 실제 Pod 실행  |
| CRD 등록  | 네트워크, 볼륨 |
| CRUD 동작 | 스케줄링       |

### suite_test.go -- 환경 설정

```go
// 테스트 시작 전: envtest 환경을 띄운다
var _ = BeforeSuite(func() {
    // 1. 로거 설정
    logf.SetLogger(zap.New(zap.WriteTo(GinkgoWriter), zap.UseDevMode(true)))

    // 2. envtest 환경 생성 (가짜 API 서버 + CRD 로드)
    testEnv = &envtest.Environment{
        CRDDirectoryPaths:     []string{filepath.Join("..", "..", "config", "crd", "bases")},
        ErrorIfCRDPathMissing: true,
    }

    // 3. 환경 시작 → API 서버 주소를 받아온다
    cfg, err = testEnv.Start()

    // 4. 클라이언트 생성
    k8sClient, err = client.New(cfg, client.Options{Scheme: scheme.Scheme})
})

// 테스트 끝: 환경 정리
var _ = AfterSuite(func() {
    cancel()
    testEnv.Stop()
})
```

### simpleapp_controller_test.go -- 테스트 케이스

```go
var _ = Describe("SimpleApp Controller", func() {
    Context("When reconciling a resource", func() {

        BeforeEach(func() {
            // Spec을 채워서 CR 생성 (image, replicas 필수)
            replicas := int32(1)
            resource := &appsv1.SimpleApp{
                ObjectMeta: metav1.ObjectMeta{
                    Name:      "test-resource",
                    Namespace: "default",
                },
                Spec: appsv1.SimpleAppSpec{
                    Image:    "nginx:latest",
                    Replicas: &replicas,
                },
            }
            Expect(k8sClient.Create(ctx, resource)).To(Succeed())
        })

        AfterEach(func() {
            // Deployment 정리 (Reconcile이 생성했을 수 있음)
            deploy := &k8sappsv1.Deployment{}
            err := k8sClient.Get(ctx, typeNamespacedName, deploy)
            if err == nil {
                Expect(k8sClient.Delete(ctx, deploy)).To(Succeed())
            }

            // SimpleApp 삭제
            resource := &appsv1.SimpleApp{}
            err = k8sClient.Get(ctx, typeNamespacedName, resource)
            if err != nil {
                return
            }
            Expect(k8sClient.Delete(ctx, resource)).To(Succeed())

            // Reconcile 호출하여 Finalizer 정리 → CR이 실제로 삭제됨
            controllerReconciler := &SimpleAppReconciler{
                Client: k8sClient, Scheme: k8sClient.Scheme(),
            }
            controllerReconciler.Reconcile(ctx, reconcile.Request{
                NamespacedName: typeNamespacedName,
            })
        })

        // 테스트 1: Reconcile이 에러 없이 완료되는지
        It("should successfully reconcile the resource", func() {
            controllerReconciler := &SimpleAppReconciler{
                Client: k8sClient,
                Scheme: k8sClient.Scheme(),
            }
            _, err := controllerReconciler.Reconcile(ctx, reconcile.Request{
                NamespacedName: typeNamespacedName,
            })
            Expect(err).NotTo(HaveOccurred())
        })

        // 테스트 2: Reconcile 후 Deployment가 생성되었는지
        It("should create a Deployment after reconcile", func() {
            controllerReconciler := &SimpleAppReconciler{
                Client: k8sClient,
                Scheme: k8sClient.Scheme(),
            }
            _, err := controllerReconciler.Reconcile(ctx, reconcile.Request{
                NamespacedName: typeNamespacedName,
            })
            Expect(err).NotTo(HaveOccurred())

            // Deployment가 생성되었는지 확인
            deploy := &k8sappsv1.Deployment{}
            err = k8sClient.Get(ctx, typeNamespacedName, deploy)
            Expect(err).NotTo(HaveOccurred())

            // Deployment 이름과 Container 수 검증
            Expect(deploy.Name).To(Equal("test-resource"))
            Expect(deploy.Spec.Template.Spec.Containers).To(HaveLen(1))
        })
    })
})
```

관련 커밋: `0ee757b` (Deployment 생성 검증 테스트 추가, Finalizer 정리 로직 수정)

핵심 포인트:

1. **BeforeEach에서 Spec을 채워야 한다** -- image가 비어있으면 Deployment 생성이 의미 없음
2. **AfterEach에서 Finalizer를 처리해야 한다** -- Reconcile이 Finalizer를 추가하기 때문에,
   CR을 삭제한 후 Reconcile을 한 번 더 호출해서 Finalizer를 제거해야 CR이 실제로 삭제됨
3. **Reconcile 함수를 직접 호출**해서 결과를 검증한다 -- 실제 Manager를 띄우지 않는다

### 실행 방법

```bash
# envtest 바이너리 설치 (최초 1회)
make setup-envtest

# Integration 테스트 실행
make test
```

## 3. E2E 테스트

### 우리 프로젝트의 파일

```text
test/e2e/
    e2e_suite_test.go  -- E2E 테스트 스위트 설정
    e2e_test.go        -- 실제 E2E 테스트
```

### 특징

- `//go:build e2e` 태그가 있어서 일반 `go test`로는 실행되지 않는다
- **진짜 클러스터**에 CRD 설치, Operator 배포, 전체 동작을 검증한다
- `kubectl` 명령어를 직접 실행한다

### 테스트 흐름

```text
BeforeAll:
    1. kubectl create ns myoperator-system
    2. make install         (CRD 설치)
    3. make deploy          (Operator 배포)

테스트:
    4. Operator Pod가 Running인지 확인
    5. 메트릭 엔드포인트가 동작하는지 확인
    6. cert-manager가 인증서를 생성했는지 확인
    7. Webhook CA가 주입되었는지 확인

AfterAll:
    8. make undeploy        (Operator 제거)
    9. make uninstall       (CRD 제거)
   10. kubectl delete ns    (Namespace 제거)
```

### 실행 방법

```bash
# E2E 테스트 실행 (진짜 클러스터 필요)
make test-e2e
```

## 테스트 비교

| | Unit | Integration (envtest) | E2E |
|--|------|----------------------|-----|
| 대상 | 헬퍼 함수, Webhook | Reconcile 전체 흐름 | 배포 + 전체 동작 |
| 클러스터 필요? | 아니오 | 아니오 (가짜 API 서버) | 예 (진짜 클러스터) |
| 속도 | 가장 빠름 | 빠름 (초 단위) | 느림 (분 단위) |
| CI에서 | 항상 실행 | 항상 실행 | 선택적 실행 |
| make 명령 | `make test` | `make test` | `make test-e2e` |

---

## 테스트 피라미드

```text
        /\
       /  \         E2E (적게, 느리지만 확실)
      / E2E\
     /------\
    /        \      Integration (중간)
   /Integration\
  /------------\
 /              \   Unit (많이, 빠르게)
/   Unit Tests   \
------------------
```

- Unit: 가장 많이, 가장 빠르게 (헬퍼 함수, Webhook 함수)
- Integration: 핵심 흐름 검증 (envtest로 Reconcile 테스트)
- E2E: 배포 전 최종 확인 (진짜 클러스터에서 전체 동작)

---

## 실행 명령어 정리

```bash
# Unit + Integration 테스트 (클러스터 불필요)
make test

# E2E 테스트 (진짜 클러스터 필요)
make test-e2e

# 특정 테스트만 실행
go test ./internal/controller/... -v

# Ginkgo로 직접 실행 (더 상세한 출력)
ginkgo -v ./internal/controller/...
```

## 정리

```text
make test (로컬에서 빠르게)
├── Unit 테스트
│   ├── 헬퍼 함수 (buildDeployment 등)
│   └── Webhook 함수 (Defaulter/Validator)
└── Integration 테스트 (envtest)
    └── Reconcile 함수를 직접 호출해서 검증

make test-e2e (클러스터에서 확실하게)
└── E2E 테스트
    └── 진짜 배포 후 kubectl로 전체 동작 검증
```

핵심: kubebuilder가 Integration과 E2E 테스트 파일을 scaffolding해 준다.
개발자는 TODO 주석 부분에 자기 로직에 맞는 테스트를 채워넣으면 된다.
