# 09-1. Mutating Webhook -- Automatic Default Values

## What Is a Mutating Webhook?

An HTTP endpoint that is called **before the API Server stores a CR in etcd** when the CR is created or updated.
It intercepts the request, automatically modifies (mutates) values, and returns the result.

Analogy: When you submit a form, the clerk fills in any missing fields for you.

```text
User submits CR create request
  -> API Server calls the Mutating Webhook
  -> Webhook: "replicas is empty? I'll set it to 3."
  -> The modified CR proceeds to the next step (Validating Webhook -> etcd storage)
```

## When to Use It

| Use case            | Example                                                          |
| ------------------- | ---------------------------------------------------------------- |
| Setting defaults    | Set replicas to 3 if not specified                               |
| Auto-adding labels  | Add `managed-by: myoperator` label to every CR                   |
| Sidecar injection   | Automatically add logging/monitoring containers to Pods (Istio uses this approach) |
| Value normalization | Auto-append `:latest` if an image name has no tag                |

## Difference from the kubebuilder:default Marker

The `+kubebuilder:default=1` marker also sets default values.
How is it different from a Mutating Webhook?

|                    | kubebuilder:default marker                  | Mutating Webhook                                         |
| ------------------ | ------------------------------------------- | -------------------------------------------------------- |
| Execution location | CRD schema (built into API Server)          | External HTTP server                                     |
| Configurable values | Fixed values only (`default=1`, `default="nginx"`) | Dynamic computation possible (current time, other resource references, etc.) |
| Conditional logic  | Not possible                                | Possible ("if production NS, replicas=3; if dev, replicas=1") |

Fixed defaults are fine with the marker. Use a Mutating Webhook when you need dynamic or conditional defaults.

## Is the Webhook a Separate Project?

No. It lives inside the same Operator project.

```text
myoperator/
  api/v1/
    simpleapp_types.go          <- CRD definition (already exists)
    simpleapp_webhook.go        <- Webhook logic (kubebuilder generates it here)
  internal/controller/
    simpleapp_controller.go     <- Reconcile logic (already exists)
```

The Controller and Webhook run simultaneously in the same binary:

```text
Controller Pod (a single process)
  |- Controller Manager: runs the Reconcile loop
  |- Webhook Server: listens on HTTPS endpoint (port 9443)
```

## How to Implement

Generate the webhook scaffold with kubebuilder:

```bash
# Mutating only
kubebuilder create webhook --group apps --version v1 --kind SimpleApp --defaulting

# Both Mutating and Validating
kubebuilder create webhook --group apps --version v1 --kind SimpleApp --defaulting --programmatic-validation
```

Note: For `--group`, use only the group name without the domain.
The domain is defined separately in the PROJECT file as `domain: example.com`.
(Use `apps`, not `apps.example.com`)

The `--defaulting` flag creates the Mutating Webhook.
The `--programmatic-validation` flag creates the Validating Webhook.

## Why the Webhook Requires HTTPS

The API Server communicates with the Webhook over HTTPS.
This means certificates are required, and cert-manager is typically used to manage them.

```text
API Server --HTTPS--> Webhook Server (runs inside the Controller Pod)
                      Certificate required!
```

During local development, `make run` automatically generates a self-signed certificate.
