# 01-1. Understanding a CRD File Line by Line

## First: Native Resources vs Custom Resources

Kubernetes has two types of resources.

|               | Native Resources                     | Custom Resources                       |
| ------------- | ------------------------------------ | -------------------------------------- |
| Created by    | Built into Kubernetes                | Defined and added by users             |
| Examples      | Pod, Service, Deployment             | Our Snack, SimpleApp                   |
| Structure     | Predefined                           | Defined by the user via schema         |
| Purpose       | Core Kubernetes functionality        | Domain-specific features (DB, certs)   |

Analogy:

- Native Resources = Existing facilities at a school (classrooms, cafeteria, library)
- Custom Resources = New facilities we created (snack request counter)

**A CRD is the "registration form" for creating Custom Resources.**
Once a CRD is registered, you can create Custom Resources of that kind.

---

Let's examine the `examples/crd.yaml` file line by line.

## Overall Structure

A CRD file has 3 main parts:

```text
1. What this file itself is (apiVersion, kind, metadata)
2. Name and scope of the new resource (group, names, scope)
3. Shape of the new resource (schema)
```

## Part 1: What This File Is

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: snacks.school.example.com
```

### apiVersion: apiextensions.k8s.io/v1

This means "this YAML uses Kubernetes' CRD feature."
This is a built-in Kubernetes API. We don't create it.
It's always `apiextensions.k8s.io/v1`. No other option (fixed value).

Every Kubernetes YAML has an apiVersion:

- Pod: `v1`
- Deployment: `apps/v1`
- CRD: `apiextensions.k8s.io/v1`

### kind: CustomResourceDefinition

This means "this YAML registers a CRD."
Same as writing `kind: Deployment` when creating a Deployment.

### metadata.name: snacks.school.example.com

The name of the CRD itself. A unique identifier for this CRD within Kubernetes.
**Must follow the `{plural}.{group}` format.**

```text
snacks.school.example.com
^^^^^^ ^^^^^^^^^^^^^^^^^^
plural   group
```

- `snacks` = the value defined in `spec.names.plural` below
- `school.example.com` = the value defined in `spec.group` below

If you choose an arbitrary name (e.g., `my-snack-crd`), Kubernetes will reject the CRD registration.

## Part 2: Name and Scope of the New Resource

```yaml
spec:
  group: school.example.com
  names:
    kind: Snack
    plural: snacks
    singular: snack
  scope: Namespaced
```

### group: school.example.com

The API group name. It goes into the `apiVersion` when creating a CR:

```yaml
apiVersion: school.example.com/v1
  ^^^^^^^^^^^^^^^^^^
  this is the group
```

You don't need to own the actual domain. It serves as a namespace to prevent name collisions.

### names

| Field    | Value  | Where it's used                                  |
| -------- | ------ | ------------------------------------------------ |
| kind     | Snack  | In CR YAML: `kind: Snack`                        |
| plural   | snacks | In kubectl: `kubectl get snacks`                 |
| singular | snack  | In kubectl: `kubectl get snack` (singular works) |

kind uses PascalCase (capitalized), plural/singular are lowercase.

### scope: Namespaced

Whether this resource belongs to a namespace.

- `Namespaced`: Belongs to a specific namespace like `default`, `kube-system` (most resources)
- `Cluster`: One per cluster. No namespace (like Node, PersistentVolume)

## Part 3: Shape of the New Resource (Schema)

```yaml
versions:
  - name: v1
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            properties:
              menu:
                type: string
              quantity:
                type: integer
```

### versions

**Note: There are TWO "versions" in a CRD file. Don't confuse them.**

| Which version             | Location          | Who defines it   | Value                                    |
| ------------------------- | ----------------- | ---------------- | ---------------------------------------- |
| `apiextensions.k8s.io/v1` | Top apiVersion    | Kubernetes       | Always `v1` (fixed)                      |
| `spec.versions[].name`    | Here              | Us               | `v1alpha1`, `v1beta1`, `v1`, etc. (free) |

The top apiVersion is "the version of the CRD feature itself," while versions here is "the version of the resource we're creating."

A single CRD can have multiple versions (v1alpha1, v1beta1, v1, etc.).
For now, we only have v1.

- `served: true` -- This version can be used via the API
- `storage: true` -- This version is stored in etcd (only one version can be true)

### schema (openAPIV3Schema)

**Defines "what fields can go in this resource's spec."**

```yaml
spec:          # The spec part of the CR
  properties:
    menu:       # Field name: menu
      type: string    # Type: string
    quantity:   # Field name: quantity
      type: integer   # Type: integer
```

Thanks to this, when creating a CR:

```yaml
spec:
  menu: chocopie    # string, so OK
  quantity: 30      # integer, so OK
  color: red        # not in schema -- ignored
```

Invalid types are rejected:

```yaml
spec:
  quantity: "thirty"    # integer expected but got string -- error!
```

## CRD and CR Relationship at a Glance

```text
CRD (crd.yaml)                    CR (cr.yaml)
--------------                    ------------
group: school.example.com    -->  apiVersion: school.example.com/v1
kind: Snack                  -->  kind: Snack
schema:
  menu: string               -->  spec:
  quantity: integer           -->    menu: chocopie
                                     quantity: 30
```

The CRD is the "mold," and the CR is "data shaped by that mold."
