# 모니터링과 로깅

## 한 줄 요약

controller-runtime은 로깅(zap), 메트릭(Prometheus), 헬스체크를 기본 내장하고 있어서
개발자가 별도 설정 없이 바로 사용할 수 있다.

---

## 1. 로깅

### 로깅 라이브러리: zap

controller-runtime은 Go의 `logr` 인터페이스를 사용하고,
기본 구현체로 **zap** 라이브러리를 쓴다.

`cmd/main.go`에서 설정하는 부분:

```go
opts := zap.Options{
    Development: true,   // 개발 모드: 사람이 읽기 쉬운 형식
}
ctrl.SetLogger(zap.New(zap.UseFlagOptions(&opts)))
```

| 모드                 | 출력 형식               | 용도                            |
| -------------------- | ----------------------- | ------------------------------- |
| `Development: true`  | 사람이 읽기 쉬운 텍스트 | 로컬 개발                       |
| `Development: false` | JSON (구조화된 로그)    | 운영 환경 (ELK, Loki 등에 적합) |

### 로거 사용법

Controller에서 로거를 가져오는 방법:

```go
log := logf.FromContext(ctx)   // context에서 logger 가져오기
```

`FromContext`로 가져오면 **자동으로 CR 정보(name, namespace)가 포함**된다.
별도로 넣지 않아도 어떤 리소스에 대한 로그인지 알 수 있다.

### 로그 레벨

```go
// Info -- 정상 동작 기록
log.Info("Deployment 생성", "name", app.Name)

// Error -- 에러 기록 (영구적 에러에 직접 사용)
log.Error(err, "외부 API 호출 실패", "url", apiURL)

// Debug -- V(1) 이상으로 설정 (기본적으로 출력 안 됨)
log.V(1).Info("상세 디버그 정보", "key", value)
```

| 레벨  | 함수              | 언제 쓰나                          |
| ----- | ----------------- | ---------------------------------- |
| Info  | `log.Info()`      | 주요 동작 (생성, 업데이트, 삭제)   |
| Error | `log.Error()`     | 영구적 에러 (재시도 무의미한 경우) |
| Debug | `log.V(1).Info()` | 상세 디버그 (운영에서는 안 보임)   |

### 우리 코드의 로그 사용 현황

```go
log.Info("SimpleApp 리소스가 삭제됨, 무시")                          // 59행
log.Info("Finalizer 정리 작업 실행", "name", app.Name)              // 71행
log.Info("Deployment 생성", "name", app.Name)                       // 96행
log.Info("Deployment 업데이트", "name", app.Name, "image", ..., "replicas", ...) // 130행
```

전부 `log.Info()`만 사용 중이다.
에러는 `return ctrl.Result{}, err`로 반환하면 controller-runtime이 자동으로 로그를 남긴다:

```
ERROR  Reconciler error  {"controller": "simpleapp", "name": "myapp", "error": "..."}
```

### 구조화된 로깅 (Structured Logging)

controller-runtime의 로그는 **키-값 쌍**으로 작성한다:

```go
// 좋은 예 -- 키-값 쌍으로 검색 가능
log.Info("Deployment 생성", "name", app.Name, "replicas", replicas)
// 출력: INFO Deployment 생성 {"name": "myapp", "replicas": 3}

// 나쁜 예 -- fmt.Sprintf로 문자열에 합치면 검색이 어려움
log.Info(fmt.Sprintf("Deployment %s 생성 (replicas: %d)", app.Name, replicas))
// 출력: INFO Deployment myapp 생성 (replicas: 3)
```

키-값 쌍으로 작성하면 로그 수집 시스템(ELK, Loki 등)에서 필드별 검색이 가능하다.

## 2. 메트릭 (Prometheus)

### 기본 내장 메트릭

controller-runtime은 Prometheus 형식의 메트릭을 자동으로 제공한다.
별도 코드 작성 없이 기본적으로 수집되는 메트릭:

| 메트릭                                      | 의미                              |
| ------------------------------------------- | --------------------------------- |
| `controller_runtime_reconcile_total`        | Reconcile 호출 횟수 (성공/에러별) |
| `controller_runtime_reconcile_errors_total` | Reconcile 에러 횟수               |
| `controller_runtime_reconcile_time_seconds` | Reconcile 실행 시간               |
| `workqueue_depth`                           | Work Queue에 쌓인 항목 수         |
| `workqueue_adds_total`                      | Work Queue에 추가된 항목 수       |
| `workqueue_retries_total`                   | 재시도 횟수                       |

### 메트릭 서버 설정

`cmd/main.go`에서 메트릭 서버를 설정한다:

```go
// 메트릭 바인드 주소 (기본: "0" = 비활성화)
flag.StringVar(&metricsAddr, "metrics-bind-address", "0", ...)

// Manager 생성 시 메트릭 옵션 전달
mgr, err := ctrl.NewManager(ctrl.GetConfigOrDie(), ctrl.Options{
    Metrics: metricsServerOptions,  // 메트릭 서버 설정
    ...
})
```

메트릭을 활성화하려면 실행 시 주소를 지정한다:

```bash
# HTTP로 메트릭 노출
--metrics-bind-address=:8080

# HTTPS로 메트릭 노출 (운영 환경 권장)
--metrics-bind-address=:8443 --metrics-secure=true
```

### ServiceMonitor (Prometheus가 수집하도록 설정)

`config/prometheus/monitor.yaml`에 ServiceMonitor가 정의되어 있다:

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: controller-manager-metrics-monitor
spec:
  endpoints:
    - path: /metrics
      port: https
  selector:
    matchLabels:
      control-plane: controller-manager
```

Prometheus Operator가 설치된 클러스터에서 이 ServiceMonitor를 적용하면,
Prometheus가 자동으로 메트릭을 수집한다.

### 커스텀 메트릭 추가

기본 메트릭 외에 비즈니스 메트릭을 추가할 수도 있다:

```go
import "sigs.k8s.io/controller-runtime/pkg/metrics"
import "github.com/prometheus/client_golang/prometheus"

var (
    simpleAppCount = prometheus.NewGauge(prometheus.GaugeOpts{
        Name: "simpleapp_active_count",
        Help: "현재 활성 SimpleApp 리소스 수",
    })
)

func init() {
    metrics.Registry.MustRegister(simpleAppCount)
}

// Reconcile 안에서 업데이트
simpleAppCount.Set(float64(activeCount))
```

## 3. 헬스체크

### Liveness와 Readiness

`cmd/main.go`에서 헬스체크 엔드포인트를 등록한다:

```go
// Liveness -- "프로세스가 살아있는가?"
mgr.AddHealthzCheck("healthz", healthz.Ping)

// Readiness -- "요청을 받을 준비가 되었는가?"
mgr.AddReadyzCheck("readyz", healthz.Ping)
```

기본 포트는 `:8081`:

```bash
curl http://localhost:8081/healthz   # Liveness
curl http://localhost:8081/readyz    # Readiness
```

### Kubernetes에서 사용

Deployment에 probe를 설정하면 쿠버네티스가 자동으로 체크한다:

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8081
  initialDelaySeconds: 15
  periodSeconds: 20
readinessProbe:
  httpGet:
    path: /readyz
    port: 8081
  initialDelaySeconds: 5
  periodSeconds: 10
```

| Probe     | 실패 시 동작                      |
| --------- | --------------------------------- |
| Liveness  | Pod 재시작                        |
| Readiness | Service에서 제외 (트래픽 안 보냄) |

## 4. Leader Election (리더 선출)

Operator를 여러 개 띄우면 (고가용성), 누가 실제로 Reconcile을 실행할지 정해야 한다.

```go
mgr, err := ctrl.NewManager(ctrl.GetConfigOrDie(), ctrl.Options{
    LeaderElection:   enableLeaderElection,
    LeaderElectionID: "1ad1a747.example.com",
})
```

```bash
# 리더 선출 활성화
--leader-elect=true
```

| 상태       | 동작                                      |
| ---------- | ----------------------------------------- |
| Leader     | Reconcile 실행, 메트릭 수집, Webhook 처리 |
| Non-Leader | 대기 (Leader가 죽으면 자동 승격)          |

## 정리

```
Operator 실행
├── 로깅 (zap)
│   ├── log.Info()    -- 주요 동작 기록
│   ├── log.Error()   -- 영구적 에러 기록
│   └── 자동 에러 로그 -- Reconcile err 반환 시 controller-runtime이 기록
├── 메트릭 (:8080 또는 :8443)
│   ├── 기본 메트릭   -- reconcile 횟수, 시간, 큐 깊이 등
│   └── 커스텀 메트릭 -- 비즈니스 지표 추가 가능
├── 헬스체크 (:8081)
│   ├── /healthz      -- Liveness (죽었나?)
│   └── /readyz       -- Readiness (준비됐나?)
└── Leader Election
    └── 여러 인스턴스 중 하나만 Reconcile 실행
```
